营销系统、商品系统复盘

Posted on 2019-05-23

在上家公司做过一个商品兑换功能，由于没有搭建供应链系统，所以业务逻辑相对简单。这次的负责商城这一块，完整地接触了电商系统的运行，我目前暂时还不能全部了解每一个独立系统的业务逻辑。电商里很多坑只有自己在设计系统或者在开发上线后才能知道并对此进行迭代。
电商系统主要分成一下几个模块，订单系统、商品系统、用户系统、供应商系统、营销系统，其他系统。我负责设计了商品系统、订单系统部分功能、营销系统。这些设计基本还停留在基础层。
商品系统模块：商品上下架、sku类目设置、sku设置属性。
商品的获取和上下架：公司的供应商来自于2方面，一个是api供应商，是接京东、苏宁等平台商品，通过接口获取他们sku的分类、属性、和图片进入到我们商品库，从商品库中挑选商品重新设置分类、品牌、属性上架到我们电商平台。基于这种供应商模式，从商品发货、售后处理都是通过调用api接口返回的结果来处理。另一个是入驻供应商，由供应商通过我们供应商平台上架商品，上架前同样要设置sku的分类、属性等。
商品在前端的展示目前是按照sku来展示，这样就会出现不同供应商的同一商品在用户端会出现多个，这样会极大地降低用户的搜索效率。我认为在资源充足的情况下，客户端可以按照spu展示，统一不同供应商提供的sku，给每一个商家唯一item id，每一个item id 再对应多个sku id。用2个字段去控制商品的唯一性。
sku类目设置：目前比较普遍的做法是设计3级类目。目前我们公司前后端是同一套商品类目，但是其实更通用的做法是后端设置一套类目保持不变，客户端展示的类目可以根据运营需求来变更。
sku属性设置：对商品上架需要设置属性，属性暂时部分销售属性和非销售属性，也就是全部定义为非销售属性。api供应商直接获取其他平台定义好的属性，入驻供应商需填写系统设置的属性类别，并且可自定义进行添加属性类别并属性值。由程序校验这些属性值组合起来是否唯一。
营销系统：营销组件、活动系统
营销组件、活动系统：我目前设计过的营销组件为限时购，限时购我刚开始设想的时候有2种方案。第一种，创建营销组件，关联商城专题。另外一种增加活动系统，也就是将营销组件关联活动，再讲活动关联专题。逻辑是没有太大出入，但是哪一种好我暂时没办法判断。0.0
我仔细说下第二种具体设计细节。限时购组件中包含4个表单,名称、生效时间、活动标签、库存、限时价。状态有开启、禁用、删除。创建活动中包含3个表单，名称、活动时间（时间段）、活动标签。状态有开启、禁用、删除。活动关联开启的限时购组件，专题再和活动进行关联。
