PACS系统复盘

Posted on 2020-04-02

RIS是放射信息管理系统，包括预约检查登记、出片、报告、审核等流程；PACS是影像归档和通信系统，即解决dicom医学影像存储、调用。RIS和PACS系统称为大影像系统。此项目开发的系统将完全替换集团现有系统，另外在解决内部需求的基础上，最终将产品设计成一个saas模式对外提供服务

在第一个版本上线前对自己未考虑全面的问题

第一：因医疗系统的复杂性和特殊性，在新系统上线后，不能直接替换掉原系统，需要和老系统并行一段时间，等到新系统稳定后进行全局替换。因未事先考虑此方案，导致测试环节对并行的流程测试不充分。并行方案中主要是考虑基础信息系统和设备系统进行2套数据交互的可行性，接入2套系统后各个系统之间的数据会不会混乱，后设计方案如下

  • 第一步：对接his系统，试运行期间针对从his推送的数据实行双写，将数据分别写入新老ris系统；

  • 第二步：医疗设备工作台获取ris数据形成worklist，调研发现，worklist只能接受一个服务的数据传输。在这种情况下，优先保障原系统的正常运作，新系统只能在离线状态提前测试，在业务正常运行期间依旧对接原ris系统。形成worklist后，可将生成的影像传输至新老pacs系统，新老系统同时对同一份影像进行分开存储和分发

  • 第三步：新老阅片系统分别从对应的新老pacs系统中调阅影像数据，新系统若涉及调阅上线前的影像数据，则通过数据接口的方式从原数据库中去捞旧数据。（pacs影像存储数据庞大，数据全部迁移较为困难）

    框架图：https://3up6er.axshare.com

第二：未考虑和其他外设的对接，后出具的方案如下

  • 排队叫号系统、报告打印系统通过和厂商沟通，通过数据库视图和存储过程的方式进行数据交互；

    对接流程图：https://oam3re.axshare.com

  • 扫描仪、身份证识别仪根据对方提供的标准接口接入；

第三：涉及到his、ris、pacs系统间的对接流程未事先设计完成，导致数据流通链路边长

  • 后给出接口对接方案，和his之间的检查单数据流通延续之前系统的对接方式，通过数据库视图和存储过程对接。设计到新ris和pacs系统通过API对接

    接口对接方案：https://3up6er.axshare.com

第四：旧数据迁移问题未考虑，复盘如下

  • 因新老系统业务逻辑基本一致，在开发设计架构时应让他们参考来系统数据表结构，为后面老数据迁移做准备；
  • 现累计数据过于庞大，在迁移过程中产生的新数据需要进行追加

正式上线后遇到的主要问题并进行迭代的过程

第一：因为dicom影像文件较大，并且对于影像保真度的要求教高，阅片软件在调阅影像的过程中下载速度太慢，满足不了业务需求，v1.1版本方案如下：

  • 对影像进行预加载处理，即通过阅片软件服务通过轮训的方式时时获取影像文件进行欲下载，并监测磁盘容量的进行定时清理
  • 计划后续能通过采用更好的无损压缩的算法减小带宽和服务器的压力

第二：基于医疗集团用户的特殊性，即便新系统的交互更加符合正常的操作习惯，但是让他们重新学习新系统交互的教育成本过大，后将一些界面按照原系统进行修改；

第三：后续因业务需要，在接入其他医院的检查单后，需要将内网上生成的影像和报告回传，对于提供的标准接口需要做安全设计，v1.1方案

  • ip加白
  • 参数加密

开放平台设计

Posted on 2020-01-14

产品是基于saas服务的crm系统，saas服务往往只能满足b端用户的共性需求，后期虽然可以通过迭代满足其他个性化需求，但是因底层数据结构以及产品结构的限制，开发成本会逐渐递增，本期产品决定进行一次大的改版，由saas向paas化过度。我负责开放平台的搭建。
整个开放平台主要有4个模块构成，开发者中心、服务市场、API网关，应用中心
开发者中心。提供给开发者入驻成为应用服务商的平台。当时我们考虑到，服务商实际上提供的是一种某个时段的某个服务，所以在应用的基础上，又包了一层服务的概念，在购买上提供了license的选择。这个服务在购买以及形态上类似于阿里云服务器以及扩容。
服务商成功入驻到开放平台后，服务商可创建应用，配置相应回调接口，ip白名单，时间通知地址，创建完成后默认分配唯一appid和appsecret。

商城系统复盘

Posted on 2019-05-23

在上家公司做过一个商品兑换功能，由于没有搭建供应链系统，所以业务逻辑相对简单。这次的负责商城这一块，完整地接触了电商系统的运行，我目前暂时还不能全部了解每一个独立系统的业务逻辑。电商里很多坑只有自己在设计系统或者在开发上线后才能知道并对此进行迭代。
电商系统主要分成一下几个模块，订单系统、商品系统、用户系统、供应商系统、营销系统，其他系统。我负责设计了商品系统、订单系统部分功能、营销系统。这些设计基本还停留在基础层，没有涉及到平台层。

  • 商品系统：商品上下架、sku类目设置、sku设置属性。
  1. 商品的获取和上下架：公司的供应商来自于2方面，一个是api供应商，是接京东、苏宁等平台商品，通过接口获取他们sku的分类、属性、和图片进入到我们商品库，从商品库中挑选商品重新设置分类、品牌、属性上架到我们电商平台。基于这种供应商模式，从商品发货、售后处理都是通过调用api接口返回的结果来处理。另一个是入驻供应商，由供应商通过我们供应商平台上架商品，上架前同样要设置sku的分类、属性等。
  2. 商品在前端的展示目前是按照sku来展示，这样就会出现不同供应商的同一商品在用户端会出现多个，这样会极大地降低用户的搜索效率。我认为在资源充足的情况下，客户端可以按照spu展示，统一不同供应商提供的sku，给每一个商家唯一item id，每一个item id 再对应多个sku id。用2个字段去控制商品的唯一性。
    sku类目设置：目前比较普遍的做法是设计3级类目。目前我们公司前后端是同一套商品类目，但是其实更通用的做法是后端设置一套类目保持不变，客户端展示的类目可以根据运营需求来变更。
    sku属性设置：对商品上架需要设置属性，属性按现在成熟的电商逻辑来分是有销售属性和非销售属性，但是我们平台在供应商和用户不多的情况下全部定义为非销售属性。api供应商直接获取其他平台定义好的属性，入驻供应商需填写系统设置的属性类别，并且可自定义进行添加属性类别并属性值。由程序校验这些属性值组合起来是否唯一。
  • 营销系统：营销组件、活动系统
    营销组件、活动系统：我目前设计过的营销组件为限时购，限时购我刚开始设想的时候有2种方案。第一种，创建营销组件，关联商城专题。另外一种增加活动系统，也就是将营销组件关联活动，再将活动关联专题。逻辑是没有太大出入，但是哪一种好我暂时没办法判断。0.0
    我仔细说下第二种具体设计细节。限时购组件中包含4个表单,名称、生效时间、活动标签、库存、限时价，另外需要挑选商品。状态有开启、禁用、删除。创建活动中包含3个表单，名称、活动时间（时间段）、活动标签。状态有开启、禁用、删除。活动关联在生效期内开启的限时购组件，专题再和活动进行关联。这其中需要仔细考虑的问题有：
  1. 商品上下架对客户端限时购展示的影响；
  2. 限购数量在加购的限制，在购物车中和商品详情页下单数量的限制。
  3. 创建限时购的时候挑选商品中对已参与同类活动的商品的限制；
  4. 创建限时购的时候挑选的商品不能参加有交叉时间段的不同限时购；
  5. 限时购组件创建后编辑删除对客户端展示的影响；
  6. 限时购标签在商品展示页、购物车页、商品详情页和商品订单页的展示情况；
  • 订单管理：订单系统主要需要跑通下单逻辑，在跑通下单逻辑的基础上订单管理的模块功能自然就出来了。下单逻辑中需考虑一下细节：
  1. 订单确认页信息和实际下单信息不符的情况;
  2. 商品下架后客户端的状态；
  3. 在生成订单的过程中因供应商不同，仓库等因素需要考虑拆单的情况，这涉及到一个父订单下面需要生成2个子订单。
  • 售后逻辑：页面逻辑基本和其他电商平台一致。在设计的过程中，需考虑一下几个问题：
  1. 售后逻辑中针对订单各个状态进行售后的形式不同，订单状态按照订单状态和物流状态分为3种，待发货，待收货，已完成。针对待发货状态，售后形式为仅一种为仅退款；针对待收货状态，售后形式也只有一种仅退款，仅退款中又涉及到2种情况，物流拦截成功的情况是直接走退款流程；物流拦截不成功状态需要用户拒收后处理退款；针对已完成状态，涉及到2种情况，第一种情况为确认收货0-7天可换货、退货退款，确认手动货后操作退款流程；第二种情况确认收货7-15天只能换货，超过15天只能维修。
  2. 退货或者换货过程中涉及到退货方式，目前是2种情况，上门取件和寄件。基于前面2种状态，涉及到运费收取的问题，运费的问题设计到会员权益，需要视具体情况而定。
  3. 售后过程中需考虑对多个同种商品进行部分退款情况。
  4. 一个售后流程会在程序中相应生成一个服务单，和下单时候的订单分开处理。
  5. 在售后过程中，涉及到多个状态，经过技术评审的教训，设计的时候最好把状态表格化，例如针对商品在待收货状态时进行部分售后情况，原订单状态展示，服务单状态展示。。。需要经过仔细思考。
  6. 售后过程中会涉及到优惠的情况的处理，目前我们的处理是将优惠全额返回，这是比较粗糙的做法，对这块还需要进入深入思考。

我的第一篇blog

Posted on 2019-03-20

第一次写blog，说下我在这家公司进行除主产品设计以外的工作

  • 设计官网，思维模式从who,what,how,why，即，我们是谁，产品介绍，使用介绍，应用场景（痛点）。
    我们是一个员工激励saas平台，我们官网是给一群带有目的而来的hr或者是企业管理者展示的。展示出我们产品的亮点。达到转化率最大化，这是我做官网的初衷。

  • 官网进行产品内审的时候，得到的反馈是，不性感，重新设计。我对不性感这个词的解读是：不打动人，理念没传达到位。我开始重新审视设计的官网，如果单单是传达理念，未免显得太过空泛，用户不能一下子get到产品的点，进而降低了转化率。但是换个思维方式想，愿景即产品想要达到的效果，如果官网能体现出用户使用我们产品之后的效果是不是会比单单介绍产品本身更能刺激用户转化。后来我收集了一些企业中暖心的、进取的用户story，做为官网首页的展示。目前数据还没有出来，不知道哪种思路更好

  • 产品上线后，我出去给企业路演过2次，收到关于以下几点反馈：
    ①注重细节，包括不能有错别字，不能用不当的icon或者图片，这会让对方觉得我们很不专业；
    ②ppt的风格和产品目标人群或者企业的风格相近，例如我是去互联网科技公司宣讲，那ppt的风格就要科技感十足，这样会更具有说服力；
    ③ppt上的语言尽量精练，因为写的太多用户也并不care，反而会起到反效果；
    ④ppt的制作要基于你对自己的产品足够了解，对产品的roadmap有规划，你才能应对在宣讲过程中用户提出的各种疑问。
    我反感这些工作，刚开始是因为我觉得他不是一个产品经理应该做的工作。但是反问自己，作为一个产品经理的职责又是什么。产品经理=产品设计+产品运营+产品销售。而产品人经常会误把产品设计作为工作重心，其实不然，产品设计只是产品人应该做的最基础也是占比最小的一部分工作。
