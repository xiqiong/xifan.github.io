象加项目复盘

Posted on 2018-05-23

象加是一个服务于互联网企业的saas系统，主要满足员工福利弹性化的需求，以此为切入点，旨在提升员工对企业的满意度，帮助企业实现福利价值的最大化。

首先象加平台帮助企业搭建属于他自己的积分商城（小程序端），企业通过从象加平台购买积分发放给员工的形式实现福利的发放，员工可在积分商城中使用积分兑换自己喜欢的商品，以此来实现员工福利的弹性化。

象加的商业模式是通过赚取供应链中的差价来实现盈利

  • 业务中设计角色说明：
    • 供应商：提供商品方，此版本只接入平台供应商（包括苏宁，京东等平台）
    • 象加平台：提供弹性福利服务平台
    • 服务商（会员）：协助象加平台获取发展企业客户，并从供应链中获取差价（付费入驻象加平台）
    • b端企业客户：象加的服务对象，主要客户为互联网企业
    • c端用户：b端企业客户的员工
  • 业务中价格名词说明：
    • 成本价：象加向供应商采购商品的单价
    • 会员价：会员向象加平台采购商品的单价
    • 分销价：企业客户向会员采购商品的单价
    • 销售价：企业客户上架到积分商城的商品单价

  • 业务流说明：

    • 象加平台接入供应商后，挑选商品上架到象加商品中心，并设置商品单价
    • 象加平台从商品中心挑选商品上架到企业管理后台的商品中心，并设置商品单价
    • b端企业客户从企业管理后台挑选商品上架到积分商城
    • c端用户从积分商城下单

  • 订单流说明：

    • c端用户从积分商城下单产生订单
    • 订单推送到供应商
    • 供应商提供供应链服务
  • 资金流：
    • c端用户从积分商城使用积分下单并且确认收到商品
    • 供应商平台将会扣除象加在供应商后台充值的预充款（象加支付成本价）
    • 象加平台将会扣除会员在会员后台充值的预充款（会员支付会员价）
    • 会员将会扣除企业客户在企业后台充值的预充款（企业客户支付分销价）

象加的产品架构：

  • 应用端：企业客户的积分商城
  • 管理端：积分商城管理后台
  • 象加运维平台
  • 商品中心

我在此版本中负责企业客户应用端和管理端的规划。这2个系统在流程和逻辑上类似于电商系统，我从电商系统的角度去考虑整个产品的服务架构，主要包括供应商模块（包含供应链服务）、订单模块、商品模块、财务模块、营销模块。

供应商模块

  • 供应商商品信息通过API进行获取，因供应商繁多，象加平台对于从不同供应商同步的同一商品存储为2个sku，那么这些同一个商品在应用端的展示也将会是多个sku。这样极大的降低了商品的展示效率，也增加了用于筛选商品的成本

    方案：可人工筛选处理，从商品中心筛选的时候过滤掉同质化的商品再上架给客户

  • 同一商品在不同的供应商平台设置不同的类目

    方案：对所有同步过来的商品进行类目重设置，应用端的筛选是针对重设置后的类目进行筛选

营销模块

在做限时购时未考虑全面的情况：

  1. 限购数量在加购的限制，在购物车中和商品详情页下单数量的限制；
  2. 同一个商品同时参加不同营销活动下的策略；
  3. 同一个商品同时参加统一时段内的不同限时购活动提示；
  4. 限时购组件创建后编辑删除对客户端展示的影响；（用户在下单时查询该商品是否参与营销活动，下单之后删除无影响）
  5. 限时购标签在商品展示页、购物车页、商品详情页和商品订单页的展示情况；

订单模块

  1. 在进入详情页和下单时需校验以下几块内容：营销活动、上下架状态、库存状态，需要分别给到相应页面处理
  2. 购物车逻辑中，需要考虑离线加入购物车的情况，并且在登录后将离线加入的商品统一展示
  3. 对多个sku合并下单时，根据供应商拆分成多个子订单。支付现金和积分按照每一子订单价格进行分摊
  4. 在应用端订单中展示订单状态时，需要将订单状态和售后状态综合起来考虑，给出组合状态表

售后模块

  1. 在对某一订单进行售后处理时，重新创建售后单，不在原有订单逻辑上增加售后状态

其他

  1. 进行企业客户小程序待开发前，需要企业客户进行开发授权
